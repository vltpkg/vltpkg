vlt is the package registry built for JavaScript: a home for the
packages your team installs and publishes, engineered around the
dependency graph. A package manager with secure defaults and a CLI for
exploring that graph come with it, but the registry is the foundation
everything else builds on.

Install the same npm packages in less time, block malicious
dependencies before they reach your builds, publish privately, and
trace every relationship in your dependency graph ...all without
changing the tools your team already uses.

## A new package registry

JavaScript packages are not just artifacts. They form enormous,
dynamic dependency graphs that affect every install, build, and
deployment. Most registry infrastructure was designed to store files
instead of empowering developers to easily understand
[the relationships between them](https://www.vlt.io/blog/the-massive-hole-in-the-npm-ecosystem).

## Motivation

> We're doing this because we expect more from the systems we've
> always counted on to host and distribute our software.

vlt is built around the JavaScript dependency graph from the ground
up, so delivery, security, and observability all come from
understanding those relationships rather than bolting features onto a
file store.

## Move fast without changing your tools

vlt serves npm-compatible packages through infrastructure optimized
specifically for JavaScript package metadata and installation
patterns. You reduce download size, improve cache efficiency, and cut
CI time while continuing to use the tools your team already knows.

- **First-class support for JavaScript & TypeScript projects.**
  Whether you use Next, Vue, or Svelte, we run our benchmarks against
  real projects for those frameworks.
- **The receipts, on clean installs.** Across the benchmark suite, the
  vlt registry serves the same packages 3–6x faster than AWS
  CodeArtifact and 4–9x faster than GitHub Packages, and stays ahead
  of JFrog and the public npm registry on typical framework apps (npm:
  Astro 1.7x, Svelte 1.6x, Vue 1.25x). Numbers move daily, so see the
  [live registry benchmarks](https://benchmarks.vlt.sh/#/registries/registry-clean).
- **Slimmer packuments.** The vlt registry serves packuments 76%
  smaller than the ten thousand most downloaded npm packages, a
  problem big enough that packument bloat
  [blocked a popular library's releases for a month](https://www.vlt.io/blog/packument-size-limits).

## Stop unsafe packages before they reach your builds

Every public package request passes through a security layer that can
identify known malware and vulnerabilities before the package is
served. vlt also reduces install-time risk with safer package-manager
defaults and enforceable dependency policies.

- **Registry protection.** Known malicious packages are blocked before
  delivery, backed by continuous advisory and malware intelligence, on
  a safe npm mirror.
- **Safer installation.** Our CLI separates run and build steps
  through a two-phase install (first download, then opt-in build), so
  potentially malicious
  [lifecycle scripts](https://www.vlt.io/blog/npm-12-breaking-changes)
  are restricted by default. These are secure defaults rather than
  opt-in hardening.
- **Integrity you can trust.** Content-addressable storage reverifies
  each package on every fetch.

This matters more every release. Despite 63% download growth,
[Sonatype's State of the Software Supply Chain 2026](https://www.sonatype.com/state-of-the-software-supply-chain/introduction)
found 838,778 packages published with a CVSS 9.0+ score, and AI
assisted development only compounds the pressure when human review and
aging platforms can no longer keep up with the demands for release,
recovery, and verification.[^1]

### Verify reproducibility, not just provenance

[vlt/reproduce](https://github.com/vltpkg/reproduce) is an open source
tool that takes a different approach to trust: it downloads, rebuilds,
and compares packages against their exact source commits to confirm a
published artifact really matches its source. Not security theatre,
but [verifiable outcomes](https://www.vlt.io/blog/reproducibility).

## Understand every dependency

Explore public packages and your project's fully resolved dependency
graph. Trace why a dependency exists, inspect how specifications
resolve, identify affected transitive packages, and query the graph
with selectors designed specifically for JavaScript dependencies.

- **Explore.** Browse package and ecosystem relationships.
- **Trace.** See why and how dependencies were resolved.
- **Query.** Identify packages using Dependency Selector Syntax (DSS),
  including
  [security insights powered by Socket](https://www.vlt.io/blog/insights-selectors).
- **Control.** Turn those queries into enforceable CI rules and
  [graph transformations](https://www.vlt.io/blog/introducing-graph-modifiers),
  blocking dependencies based on package properties, relationships, or
  security metadata.

## The private registry built for JavaScript teams

[Publish scoped and private packages](https://www.vlt.io/blog/pack-publish-version),
manage organizations and access, and give every developer and CI
environment a consistent source for public and private JavaScript
dependencies.

vlt is a strong fit if:

- you or your team publish packages that other critical software
  depends on
- you serve dependencies at scale (100k+ downloads/month) and need
  them to stay fast and available
- you want risky lifecycle scripts and unvetted dependencies contained
  by default, not by opt-in hardening
- your organization struggles to keep frameworks and dependencies
  current
- you need every package and dependency accounted for, from local
  installs through CI

See how it holds up on your own dependencies.
